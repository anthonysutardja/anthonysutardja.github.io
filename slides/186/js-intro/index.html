What’s a web page?

A web page is a HTML document. These documents can contain JavaScript (abbreviated JS).

<!DOCTYPE html>
<html lang="en">
    <head>
        <title>Hi there.</title>
    </head>
    <body>
        <h1>Rick Grimes</h1>
        <!-- JavaScript block loaded through some file -->
        <script src="static/js/app.js"></script>
        <!-- JavaScript evaluated while rendering the page -->
        <script>
            var rick = 'police man';
        </script>
    </body>
</html>

JavaScript Syntax

JS syntax is extremely forgiving, which means debugging is a pain.

JS is loosely typed (sorry).

var name = 'Rick Grimes';
var zombiesKilled = 9000;
var findZombies = function(locations) { return []; };

abstract, boolean, break, byte, case, catch,

goto, if, implements, import, in, instanceof,

private, protected, public, return, short, static,

while, with

But most of these aren’t implemented in JS… (will work in chrome but don’t do it!)

JS basic types

var a = 1;                // number
var b = 1e+10;            // number
var c = 1.01;             // number
var d = true;             // boolean
var d = 'hi';             // string

And there are others but they are just objects.. (like Array)

Convert string to number

var strNum = '123';
var converted = +strNum;  // 123

Convert number to string

var num = 123;
var converted = '' + num;  // '123'

Convert object to boolean

var obj = {};
var converted = !!obj;  // true

Objects

Are just basically hash maps.

var obj = {
    'name': 'Rick Grimes',
    'occupation': 'Police man'
};

You can get/set attributes in two ways:

obj.name = 'Karl';
obj['name'] = 'Karl';

console.log(obj.name);    // 'Karl'
console.log(obj['name']);  // 'Karl'

Equality

Always use strict equality === rather than just the regular equality ==

console.log(1 == 1);  // true
console.log(1 === 1);  // true

console.log(1 == [1]);  // true
console.log(1 === [1]);  // false

console.log(0 == []);  // true
console.log(0 === []);  // false

Functions

function fcnName(parameter1, parameter2) {
    // Local scope
    var x = 1;  // accessible only to scope
    // do something
    return true;
};

Functions don’t need to have names

function (parameter1, parameter2) {
    // Local scope
    var x = 1;  // accessible only to scope
    // do something
    return true;
};

Functions don’t need to have names, but can be assigned to a variable.

Anthony likes this way because it doesn’t do crazy things.

var fcnName = function (parameter1, parameter2) {
    // Local scope
    var x = 1;  // accessible only to scope
    // do something
    return true;
};

Functions don’t have to return anything (returns undefined by default)

function noOp() {};
console.log(noOp());  // undefined

JS doesn’t have any classes, but it kind of does

var Person = function (firstName) {
    // You can use the `this` keyword
    this.firstName = firstName;
};

var bob = new Person('Bob');
console.log(bob.firstName);  // 'Bob'

Adding class methods requires you to dive into the object’s prototype.

var Person = function (firstName) {
    this.firstName = firstName;
};

Person.prototype.setName = function (firstName) {
    // You can use the `this` keyword
    this.firstName = firstName;
};

var p = new Person('Bob');
console.log(p.firstName);  // 'Bob'
p.setName('Joe');
console.log(p.firstName);  // 'Joe'

The awful parts of JavaScript

this is the worst

In the global browser context, this refers to the global context

// in your Browser's JS console
console.log(this === window);  // true

In a function? It depends..

var foo = function() {
    return this;
};

/** some code in between **/

// should be the global object
console.log(foo());
// but not necessarily..

Things get complicated in callbacks

var f = function () {
    // Run anonymous function 
    setTimeout(100, function () {console.log(this);})
};

f();  // 100 ms later --> "undefined" is logged

It’s straight forward in “classes” right?

var Incrementer = function () {
    this.inc = 1;
};

Incrementer.prototype.increment = function(number) {
    return this.inc + number;
};

var i = new Incrementer();
console.log(i.increment(2));  // 3

Nope. Special JS methods can change what this is

var Incrementer = function () {
    this.inc = 1;
};

Incrementer.prototype.increment = function(number) {
    return this.inc + number;
};

var i = new Incrementer();
console.log(i.increment(2));  // 3

var obj = {'inc': 2};

// Binds `this` to `obj` one time
console.log(i.increment.call(obj, 2));  // 4
// Binds `this` to `obj` one time
console.log(i.increment.apply(obj, [2]));  // 4

And another way…

var Incrementer = function () {
    this.inc = 1;
};

Incrementer.prototype.increment = function(number) {
    return this.inc + number;
};

var i = new Incrementer();
console.log(i.increment(2));  // 3

var obj = {'inc': 2};

// Binds `this` to `obj` in a new function with the same scoping as the original function
var func = i.increment.bind(obj);
console.log(func(2));  // 4

When using frameworks, all sorts of crazy magic is happening and you don’t have any way of knowing about it. You need to be careful regarding what this is. If you’re interested, you can read more about this on your own time.

Classic work arounds: use bind

var Person = function (firstName) {
    this.firstName = firstName;
};

Person.prototype.getName = function () {
    return this.firstName;
}

var b = new Person('Bob');

/** bunch of code that does something mysterious **/

setTimeout(1000, b.getName.bind(b));  // guarantee binding to `b`

Another hack

Rebound methods still have the same scope… let’s use that.

var Person = function (firstName) {
    this.firstName = firstName;
};

Person.prototype.delayedShout = function () {
    var that = this;
    setTimeout(1000, function() {
        console.log(that.name);
    });
}

/** bunch of code that does something mysterious things in 1 second **/

Other awful parts of JavaScript

What is a after execution?

var a = 10;

function b() {
    a = 100;
    if (true) {
        return;
    } else {
        var a = 1000;
    }
};

b()
console.log(a);

a is obviously still 10

var a = 10;

function b() {
    a = 100;
    if (true) {
        return;
    } else {
        var a = 1000;
    }
};

b()
console.log(a);  // 10

Hoisting (a horrible feature of JS)

Function declarations and variable declarations (not assignments) are hoisted magically to the top of the containing scope.

function golden() {
    bears();
    var x = 1;
};

Is really ..

function golden() {
    var x;  // declarations are hoisted
    bears();
    x = 1;
};

What’s entered into the JS interpreter:

var a = 10;

function b() {
    a = 100;
    if (true) {
        return;
    } else {
        var a = 1000;
    }
};

b()

What’s actually interpretted:

var a = 10;

function b() {
    var a;
    // assignment to locally scoped `a`
    a = 100;
    if (true) {
        return;
    } else {
        // Declaration hoisted, but not assignment
        a = 1000;
    }
};

b()
// `a` out here is unmodified and still 10

Be careful when declaring functions and variables.

JavaScript’s syntax is forgiving, but it doesn’t mean you should code carelessly.
